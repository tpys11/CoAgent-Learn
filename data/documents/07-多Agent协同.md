# 多Agent协同架构

## 为什么需要多Agent

单个Agent要同时完成"理解用户水平 + 检索知识 + 写出好内容 + 检查错误"，任务太复杂。拆成多个专业Agent各有分工，效果更好：

- 每个Agent只专注一件事，做得好
- Agent之间可以相互检查、纠错
- 架构清晰，容易维护和扩展

## 六种通信模式

### 1. A2A（Agent to Agent —— 单向委托）
主Agent把子Agent当工具调用。传入参数，拿回结果。简单直接，适合顺序任务。

### 2. 黑板模式（Blackboard / Shared State）
所有Agent读写同一个共享数据区，不直接通信。某个Agent产出写入→状态变化触发后续Agent→读取新状态→执行。

### 3. 编排器模式（Orchestrator）
一个中心化的程序（不一定是Agent）控制所有Agent的执行顺序、参数传递和结果回收。Agent之间完全隔离。

### 4. 发布-订阅模式（Pub/Sub）
Agent通过事件总线通信。Agent可以发布事件（"数据清洗完成"），也可以订阅自己关心的事件。事件触发时自动启动。

### 5. 任务队列模式
顶层规划模块拆分任务→放入公共队列→Worker Agent空闲时领取→完成后放入结果队列。Agent之间不直接交互。

### 6. 人机在环模式（Human-in-the-Loop）
所有关键节点的输出必须先提交给人类审核，确认/修改/驳回后再流转给下一个Agent。

## 本项目选择的模式

用 LangGraph 的 StateGraph 实现**编排器模式**：三个Agent在一个共享State上以流水线方式传递数据，审核Agent可以条件路由回生成Agent（重试），整体由编排器控制节奏。

## 本项目的三个Agent

| Agent | 职责 | 输入 | 输出 |
|-------|------|------|------|
| 学情诊断Agent | 评估用户知识水平 | 用户自由文本 | 结构化知识画像JSON |
| 知识生成Agent | 生成个性化学习内容 | 用户画像+检索结果 | 学习材料+引用 |
| 审核裁判Agent | 交叉验证降幻觉 | 生成Agent的完整输出 | 事实核查+修正后内容 |

审核Agent如果发现幻觉率 > 5%，会自动路由回生成Agent重新生成（最多2轮）。
