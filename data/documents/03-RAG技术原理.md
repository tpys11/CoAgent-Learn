# RAG（检索增强生成）技术

## 为什么需要RAG

大语言模型有两个先天问题：
1. 训练数据有截止日期，不知道最新信息
2. 容易产生"幻觉"——对于不懂的内容，编造看似合理的答案

RAG 就是要解决这两个问题。它的核心思想是：让模型在回答之前，先从一个外部知识库里找到相关的真实资料，再基于这些资料来回答。

## RAG 的五个步骤

### 1. 文档预处理（Parse）
把原始文件（PDF、Word、网页等）转换为纯文本，去掉格式混乱的内容。

### 2. 文档切块（Chunking）
把长文档切成一个个小片段（通常是200-800字），因为完整的文档太长，直接对比相似度不精准。好的切分策略会让检索结果质量翻倍。

常见的切分方式：
- 按标题切分（保持语义完整）
- 按固定字数切分（简单但可能破坏语义）
- 按段落切分（自然边界）

### 3. 向量嵌入（Embedding）
把每个文本片段转换成一串数字（向量），语义相近的文字在向量空间中距离很近。

同一个 Embedding 模型也要用来"向量化"用户的问题，这样才能在同一个向量空间里做对比。

### 4. 存入向量数据库
把向量和对应的原始文本一起存入向量数据库（如 Chroma、Milvus），建立索引以便快速检索。

### 5. 检索 + 生成（Retrieve + Generate）
用户提问时：
1. 将问题用同一个 Embedding 模型转为向量
2. 在向量库里找到最相似的 Top-K 个文本片段
3. 把这些片段和用户问题一起拼接成 Prompt
4. 发给大模型，让它基于提供的资料回答问题

## RAG 的关键质量因素

- 切块大小：太小丢失上下文，太大检索不准
- Embedding 模型质量：直接影响语义匹配精度
- Top-K 值：太少覆盖不全，太多会淹没关键信息
- 检索阈值：相似度低于某个值的片段直接丢弃

## 进阶：Agentic RAG

基本的 RAG 是"检索一次→回答"，Agentic RAG 则让 Agent 自己决定"什么时候需要检索、检索什么、检索结果好不好、要不要换关键词再检索"。这是从被动检索到主动信息获取的升级。
